03. Platforms

The Tridion Sites MCP Server

Building a Model Context Protocol (MCP) server for an enterprise CMS reveals unique challenges. Exposing over 80 specialized tools allows an LLM to navigate complex business rules, but requires careful architecture to avoid overloading the system prompt or context window.

Deep Dive: Enterprise Patterns →

Enterprise MCP Design Patterns

Rich Documentation

Semantic naming, detailed descriptions, and concrete examples are essential for helping agents infer hidden business rules and complex enterprise logic.

Find-Then-Fetch

Returning raw REST API responses quickly exhausts context limits. Adopting an API that requires LLMs to request specific properties ensures responses stay lean.

Bulk Orchestration

To update or classify thousands of items without spamming the client or suffering latency, a dedicated orchestration tool executes scripts securely on the server.

Architectural Approaches

Providing a large set of richly documented tools creates a massive system prompt footprint. I have explored three repository patterns to optimize tool hydration for LLM agents:

  • Standard Architecture: The baseline approach, registering the entire toolset natively on initialization.
    View Repository →
  • Call-Tool Proxy: A flexible approach exposing only two native tools (getToolDetails and callTool), acting as an on-demand gateway.
    View Repository →
  • Dynamic Registration: A cleaner architecture using a single _setTools tool combined with the MCP list_changed notification to register native tools dynamically.
