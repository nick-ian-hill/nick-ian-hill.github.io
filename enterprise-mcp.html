Mastering Enterprise MCP

Architecting Scalable AI Integrations for Complex Systems.

Building a Model Context Protocol (MCP) server for a personal project is straightforward. Building one for an enterprise system like Tridion Sites – with over 80 discrete tools, complex inheritance models (BluePrinting), and strict governance – is an entirely different engineering challenge.

The following five patterns have emerged from my work as "portable" solutions for any developer or organization facing the information density and scale limits of modern LLMs.

Pattern 01

The Semantic Bridge

In enterprise systems, business rules are rarely explicit in the code. A "Delete Page" operation might have hidden side effects like unpublishing dependencies or breaking localizations. AI agents cannot guess these rules; they must be taught them through high-fidelity tool documentation.

By providing concrete examples and detailed parameter descriptions, we bridge the gap between "System Prompt" and "Business Logic," preventing hallucinations in high-stakes environments.

Pattern 02

Find-Then-Fetch

Standard REST APIs are often "chatty" and return massive JSON payloads. In an MCP environment, returning an entire object graph quickly exhausts the LLM’s context window and inflates token costs.

"The goal isn't to give the AI all the data, but the specific data required for the next step."

We solve this by implementing an includeProperties array in fetch tools. This allows the agent to perform an initial discovery (search) and then selectively retrieve only the fields it needs (e.g., just the title and publishDate) for its current task.

Pattern 03

Server-Side Orchestration

If an agent needs to audit 5,000 pages, calling 5,000 discrete MCP tools from a client (like Claude or Gemini) creates massive latency and cost. The round-trip overhead becomes the bottleneck.

My solution is a Tool Orchestrator that functions as a server-side script engine. The LLM generates a Javascript-based script that executes directly on the MCP server. This allows for high-frequency execution loops without flooding the network or the context window.

Pattern 04

Embedded AI Intelligence

While server-side scripts are fast, they are often "dumb" loops. To handle complex semantic tasks, like auto-classifying content or generating alt text for images, the loop needs intelligence.

I have pioneered the use of AI-enabled tools within the orchestrator. This allows the server-side script to call another LLM instance (via tools like autoClassifyItem) for every iteration of the loop, providing semantic judgment at massive scale without returning control to the primary agent.

Pattern 05

Dynamic Hydration

Advertising 80+ tools in a single system prompt is an "anti-pattern" for enterprise scale. It wastes thousands of tokens and confuses the agent’s reasoning capability.

I’ve explored two "Just-In-Time" (JIT) methods to manage this:

  • Call-Tool Proxy: A gateway where only two tools (getToolDetails and callTool) are exposed natively.
  • _setTools Protocol: A mechanism that dynamically registers and unregisters native MCP tools based on the current task, using the list_changed notification.

Conclusion

The Tridion Sites MCP implementation serves as a blueprint for any enterprise system looking to become "Agent Friendly." By focusing on semantic clarity, context optimization, and server-side execution, we turn complex repositories into intelligent ecosystems.
